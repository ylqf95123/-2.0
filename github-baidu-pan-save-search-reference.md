# GitHub 参考检索：百度网盘分享页保存时搜索目标文件夹

## 结论

截至本次检索，**没有找到一个完全匹配**你当前目标的 GitHub 开源浏览器插件，即：

- 运行在 `pan.baidu.com/s/*` 或 `pan.baidu.com/share/*`
- 在“保存到网盘”弹窗里
- 允许直接搜索**自己网盘中的目标文件夹**
- 然后继续完成保存/转存

GitHub 上更常见的是下面几类项目：

1. 百度网盘下载增强脚本
2. 百度网盘分享内容批量转存脚本
3. 网盘目录/文件导出工具
4. 基于 API 或后端服务的“搜索 + 转存”系统

所以，这份文档主要提供的是**最接近的参考实现**和**可借鉴的设计点**，而不是现成可直接替换的成品。

## 最接近的参考项

### 1. Gollum for Baiduyun

- 链接: <https://gist.github.com/mcxiaoke/6a9aa975d6b7ed8e4dbb>
- 类型: 早期脚本 / Gist
- 核心能力:
  - 百度云盘批量转存用户分享
  - 支持指定 `to_folder` 作为转存目标文件夹
  - 支持差分转存
  - 支持在分享主页下进入某文件夹后继续转存其下级子文件夹
- 对你项目的价值:
  - 它证明了“**分享页 -> 选择目标目录 -> 执行转存**”这条链路是有历史脚本先例的
  - 可借鉴“目标目录参数化”的思路
- 局限:
  - 不是 MV3 浏览器插件
  - 不是在现代保存弹窗里提供搜索框
  - 年代较老，依赖百度旧页面结构

### 2. NBNT: 新版百度网盘共享文件库目录导出工具

- 链接: <https://github.com/ujincn/NBNT>
- 类型: 油猴脚本
- 核心能力:
  - 导出百度网盘共享文件库目录结构
  - 导出完整文件列表
  - 支持大文件夹分页获取
  - 支持树形/制表符/Excel 输出
- 对你项目的价值:
  - 说明新版百度网盘页面里，**目录树遍历、分页获取、大目录扫描**仍然可做
  - 对“搜索前是否需要采集目录树”有参考价值
- 局限:
  - 目标是导出，不是保存弹窗搜索
  - 没有“在转存弹窗中搜索目标文件夹”的交互

### 3. bdpan-storage / baidu-drive

- 链接: <https://github.com/alexander-china/bdpan-storage>
- 原始来源: `baidu-netdisk/bdpan-storage`
- 类型: 百度网盘 AI Skill / API 驱动
- 核心能力:
  - 上传、下载、转存、分享、搜索、移动、复制、重命名、创建文件夹
  - 支持“搜索网盘里所有 PDF 文件”“转存这个链接到网盘”等指令
- 对你项目的价值:
  - 说明“**搜索我的网盘内容**”这件事更适合走**官方接口/后端能力**
  - 如果以后不坚持纯前端 DOM 方案，这类项目是最值得借鉴的方向
- 局限:
  - 不是浏览器页面增强插件
  - 不是直接改造百度分享页保存弹窗
  - 需要授权和外部能力支持

### 4. BDPan Search - 百度网盘智能搜索转存系统

- 链接: <https://github.com/cuixinqi/bdpan>
- 类型: 后端系统 + Web UI + Chrome 插件
- 核心能力:
  - 智能搜索资源
  - 一键转存
  - 历史记录
  - 系统监控
  - 自带 Chrome 插件目录
- 对你项目的价值:
  - 它很接近“**搜索 + 转存**”工作流
  - 可参考“插件只是前端入口，核心能力在后端”的架构
- 局限:
  - 搜的是资源站/资源索引，不是“保存弹窗里搜索我自己的网盘目录”
  - 更像资源管理系统，不是轻量页面增强扩展

## 对你当前插件最有价值的实现启发

### A. 如果坚持纯页面增强

最值得参考的是：

- `Gollum for Baiduyun`
- `NBNT`

可借鉴点：

- 分享页和保存链路本身是可以被脚本接管的
- 大目录/深目录需要增量扫描，而不是一次性全盘扫描
- 扫描应当围绕“当前弹窗可达目录树”进行，而不是整个网盘
- 必须处理异步渲染、分页、懒加载和 iframe/容器切换

### B. 如果允许引入接口或后端

最值得参考的是：

- `bdpan-storage`
- `BDPan Search`

可借鉴点：

- “搜索我的网盘目录”本质上更适合用 API 做，而不是全靠 DOM 猜节点
- 插件可以只做：
  - 注入搜索输入框
  - 显示搜索结果
  - 选中结果后自动写回保存目标
- 真正的目录搜索由后端或官方接口完成

## 为什么 GitHub 上很少有完全一致项目

原因通常有这几个：

1. 百度网盘前端 DOM 结构变化频繁，脚本维护成本高
2. “保存到网盘”弹窗很多时候是私有实现，不稳定
3. 直接搜索“自己网盘里的目标目录”更适合走 API，而不是 DOM
4. 这类工具用户面窄，开源实现少，很多只停留在私有脚本

## 建议你的实现路线

结合这次检索结果，你现在这个项目更适合下面两条路线之一：

### 路线 1：纯前端 DOM 增强

- 在保存弹窗中注入搜索框
- 只搜索当前目录树和已展开节点
- 当无结果时，做有限轮次的展开与滚动
- 不预扫描整个网盘

适合：

- 不想依赖任何额外服务
- 追求安装即用

缺点：

- 可靠性受百度前端结构影响大

### 路线 2：前端插件 + API 搜索

- 插件只负责 UI 注入和结果选择
- 后端或百度官方能力负责搜索用户网盘目录
- 选中后自动把目标目录回填到保存动作

适合：

- 追求真正稳定的“搜整个网盘目录”
- 愿意接入接口或本地服务

缺点：

- 实现复杂度更高
- 涉及认证和授权
