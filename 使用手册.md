# 百度网盘转存目录搜索 — 使用手册

> 一款油猴脚本(userscript),给百度网盘**分享页**的"保存到网盘"弹窗加上一个**文件夹搜索框**,让你不用一层层点开也能快速定位到深处的目录。

---

## 一、安装

### 1. 装浏览器扩展(任选其一)

| 浏览器 | 推荐扩展 | 说明 |
|---|---|---|
| Chrome / Edge | [Tampermonkey](https://chromewebstore.google.com/detail/tampermonkey/dhdgffkkebhmkfjojejmpbldmpobfkfo) | 主流选择 |
| Firefox | [Tampermonkey](https://addons.mozilla.org/firefox/addon/tampermonkey/) 或 [Violentmonkey](https://addons.mozilla.org/firefox/addon/violentmonkey/) | 都可以 |
| 国产/其他 | 自带的"油猴插件"入口即可 | 支持 GM_* API 就行 |

装完扩展并**启用**它(浏览器右上角的图标要亮着),再往下走。

### 2. 添加脚本

**方式 A:直接从本地文件打开**

1. 在 Chrome 地址栏输入 `chrome://extensions/`,确认 Tampermonkey 已启用,并**允许访问文件网址**(点扩展的"详细信息" → 打开"允许访问文件网址")
2. 双击 `baidu-pan-save-search.user.js`
3. Tampermonkey 会弹出确认页 → 点"安装"

**方式 B:复制粘贴**

1. 打开 Tampermonkey 图标 → "管理面板"
2. 顶部 `+` 号新建脚本
3. 全选删除默认内容,把 `baidu-pan-save-search.user.js` 里的**全部内容**粘进去
4. `Ctrl+S` 保存

### 3. 确认脚本已生效

1. 打开 [pan.baidu.com](https://pan.baidu.com/) 并登录
2. 点 Tampermonkey 图标 → 应该能在"已启用"列表里看到 **"百度网盘转存目录搜索"**
3. F12 → Console,过 5-10 秒后应该没有红色报错

## 二、使用流程

### 场景 1:在别人的分享页保存到自己的深层目录

1. 打开任意一个分享链接(比如 `pan.baidu.com/s/xxxx`)
2. 页面底部找到"**保存到网盘**"按钮左边的**文件夹图标**,点它
3. 弹出的目录选择框**顶部**会有一栏新加的搜索区(边框、圆角、白底,和百度原生不一样)
   ```
   ┌───────────────────────────────────────┐
   │ [搜索我的网盘文件夹...]   [刷新索引]   │
   │ 索引完成，共 128 个文件夹              │
   │ ┌────────────────────────────────┐     │
   │ │ 📁 2025-11-7收纳               │     │
   │ │    /课程/2025-11-7收纳         │     │
   │ └────────────────────────────────┘     │
   └───────────────────────────────────────┘
   ```
4. 输入关键字(比如 `收纳`)
5. 结果列表实时更新,点你要的目录 → 原生弹窗会自动定位过去
6. 手动检查目录对不对,点百度自己的**"确定"**完成转存

### 场景 2:第一次用,建索引

- 第一次装完脚本、或者刷新缓存后,搜索框会先显示:
  ```
  正在建立目录索引... 已扫描 42 个文件夹（队列剩余 17）
  ```
- 网盘目录 100 个以内:1-3 秒
- 网盘目录 1000 个左右:10-30 秒
- 建索引期间可以先干别的,索引会在**后台默默跑完**,不会阻塞你

### 场景 3:目录多了,搜索结果不理想

点搜索框右边的 **"刷新索引"** 强制重建。24 小时内不点也会用缓存;超过 24h 会自动过期重建。

## 四、配置项(需要改代码)

打开脚本文件,顶部有 `CONFIG` 对象:

```js
const CONFIG = {
  cacheKey: "baidupan_folder_index_v1",   // 缓存 key
  recentKey: "baidupan_folder_recent_v1", // 最近使用 key
  indexTTL: 24 * 60 * 60 * 1000,          // 索引过期时间:24小时
  maxDepth: 2,                            // 递归最大深度 → 索引到第 3 层
  concurrency: 4,                         // 并发请求数
  pageSize: 1000,                         // 单次拉取条目数
  debounceMs: 200,                        // 输入防抖
  maxResults: 20,                         // 最多显示几条结果
  retryCount: 2,                          // 网络失败重试次数
  retryDelayMs: 500,                      // 重试间隔
  refreshDelayMs: 120,                    // 弹窗扫描节流
  nodeSearchAttempts: 8,                  // 单段节点查找重试次数
  pathVerifyAttempts: 12,                 // 点击后路径/下级可见性校验次数
  debugReportEnabled: true,               // 是否开启本机调试日志上报
};
```

### `maxDepth` 与实际层数的换算

代码里 `maxDepth` 是"最深会**继续钻**的父目录深度",最终索引里能出现的目录**比这个数多 1 层**。

| `maxDepth` | 索引到第几层 | 例子 |
|---|---|---|
| 0 | 1 层 | `/全能视频` |
| 1 | 2 层 | `/全能视频/精武门` |
| **2**(默认) | **3 层** | **`/全能视频/精武门/精武门第二部`** |
| 3 | 4 层 | `/全能视频/精武门/精武门第二部/花絮` |
| 5 | 6 层 | ... |
| 8 | 9 层 | 基本相当于全扫 |

**收益对照**(网盘规模仅供参考):
- `maxDepth=2`(默认):100 个二级目录的网盘 → 建索引 **2-5 秒**
- `maxDepth=8`:同一网盘 → 建索引 **20-60 秒**,请求量翻十几倍

### 其它常见调整

- **搜不到深处的目录**:把 `maxDepth` 从 2 改到 3 或 4
- **想加快索引速度**:`concurrency` 从 4 改到 8(但可能触发百度限流)
- **索引太旧**:`indexTTL` 改小,比如 `6 * 60 * 60 * 1000`(6 小时)

## 五、数据保存在哪

| 内容 | 存储位置 | 换电脑是否保留 |
|---|---|---|
| 目录索引 | Tampermonkey 存储(`GM_setValue`) | ❌ 不保留 |
| 最近使用目录 | Tampermonkey 存储 | ❌ 不保留 |
| 用户 cookie / 登录态 | 浏览器自身 | 按浏览器同步策略 |

**清除数据**:
- Tampermonkey → 脚本管理 → 编辑该脚本 → 左侧标签"存储" → 手动删
- 或者代码里 `GM_deleteValue(CONFIG.cacheKey)` 跑一次

## 六、故障排查

### ❌ 打开分享页,底部点了文件夹图标,但没看到搜索框

**可能原因 1:百度改版了 DOM**

按 F12 → Console,输入:
```js
document.querySelectorAll('[class*="dialog"]').forEach(el => {
  if (el.textContent.includes('保存到')) console.log(el);
});
```
- 如果打不出东西 → 说明当前弹窗类名和脚本里配置的 `SELECTORS.dialog` 都不匹配 → 需要更新选择器
- 如果能打出但没搜索框 → 可能是 `SELECTORS.header` / `SELECTORS.body` 匹配不到,可以给作者报个反馈

**可能原因 2:Tampermonkey 没生效**

- 检查扩展是否启用
- 检查脚本是否启用(灰色开关拨成绿色)
- 分享页 URL 是否在 `@match` 范围内(`https://pan.baidu.com/s/*` 或 `/share/*`)

### ❌ 搜索框显示"目录索引失败"

- **`errno=-6`**:登录失效 → 重新登录百度网盘
- **`HTTP 403 / 401`**:cookie 有问题 → 清 cookie 重登
- **`HTTP 429`**:短时间请求太多被限流 → 等 10 分钟再试,或把 `CONFIG.concurrency` 改小
- 其他 errno → 打开 Console 看具体 API 返回,给作者提 issue

### ❌ 输入真实存在的目录名后,文字突然消失,还提示"没有匹配结果"

这类问题**通常不是百度网盘限制中文输入**,而是脚本自己的弹窗重建逻辑导致的。

**典型现象**:
- 输入一个盘里**不存在**的名字时,输入框内容能正常保留
- 输入一个盘里**真实存在**的目录名时,日志里明明已经搜到了结果,但输入框文字瞬间消失,界面又显示"没有匹配结果"

**根因**:
- 脚本原先在 `scanDialogs()` 里用 `createDialogKey()` 判断"这是不是一个新弹窗"
- 这个 key 里包含了弹窗的 `left / top / width / height`
- 当你输入真实存在的目录名后,结果列表会渲染出来,弹窗高度通常会被撑大
- 高度一变,`MutationObserver` 就会再次触发,脚本误以为这是一个"新弹窗"
- 接着又执行了一次 `injectSearchUI()`,把已经输入的搜索框和结果列表整块删掉重建

**为什么只在"有结果"时容易发生**:
- 有结果时:结果列表出现,弹窗高度变化明显,更容易触发误判
- 没结果时:结果区高度变化很小,弹窗尺寸通常不变,所以看起来输入正常

**怎么解决**:
- 不再把弹窗尺寸变化当成"新弹窗"
- 只要当前弹窗里已经存在 `.yt-pan-search-root`,就直接认为它已经注入完成,不要重复执行 `injectSearchUI()`
- 同时在找不到弹窗时再清空状态,避免旧状态干扰下次打开

**这次实际修复点**:
- 修改了 `scanDialogs()` 的逻辑
- 原来是:只有 `dialogKey === state.dialogKey` 才跳过重注入
- 现在是:只要弹窗里已经有 `.yt-pan-search-root`,就直接跳过重注入

**一句话总结**:
- 不是"中文输入被百度吃掉了",而是"搜索结果把弹窗撑高了,脚本误判成新弹窗,把输入框重建了"。

### ❌ 点搜索结果后,原生弹窗没跳过去,提示"未找到目录节点"

- **F12 Console 里会打印**当前可见的节点样例,大概长这样:
  ```
  [baidupan-search] 未找到节点 2025-11-7收纳 当前可见节点样例： ["全部文件", "2020五月份课程", ...]
  ```
- 说明脚本"看到"的树内容和"要找"的段名对不上。可能:
  - 目录被重命名过,索引里是旧名 → **点"刷新索引"**
  - 树里的名字有隐藏空格 / 特殊字符 → 需要调 `extractNodeName` 逻辑
  - 百度用了虚拟滚动,节点还没渲染 → 可以把 `findNodeInTree` 里的 `attempt` 上限调大

### ❌ 一级目录不会自动稳定展开,必须我先手动点开一级目录

**当前现象**:
- 例如目标路径是 `/全能视频/精武门`
- 脚本能搜到结果,也能尝试点击 `全能视频`
- 但百度弹窗里这个一级目录经常没有被脚本稳定展开
- 这时如果你手动点一下 `全能视频`,让它先展开,脚本再去定位二级目录通常就能成功

**这次排查到的结论**:
- 这不是单纯的"没找到一级目录",而是"点击一级目录"和"百度内部真正完成展开/切换"之间并不完全等价
- 旧日志里已经能看到:
  - 目标二级目录文字已经出现在弹窗 DOM 文本里
  - 但 `findNodeInDialog()` 的候选节点列表里,仍然可能只剩顶层目录
- 也就是说,问题更像是:
  - 一级目录点击后,百度自己的树控件没有稳定进入"已展开且子节点可用于匹配"的状态
  - 或者子节点虽然出现在 DOM 里,但没有被现有候选节点收集逻辑稳定纳入

**这次已经做的事**:
- 给定位链路补了更细的调试日志,包括:
  - 命中的节点是谁
  - 点击后 `aria-expanded` / `aria-selected` 是否变化
  - 展开按钮有没有真正命中
  - 展开前后下一级节点是否可见
- 把调试日志上报地址固定到本机 `127.0.0.1`,只用于本地排查

**当前状态**:
- 这个问题**还没有彻底修好**
- 当前版本只能更准确地判断问题卡在"点击没生效"还是"子节点没进入候选集"
- 但还不能保证一级目录一定由脚本自动展开

**临时规避办法**:
- 搜到目标目录后,如果它属于某个一级目录下面
- 先在百度原生树里手动点开那一级目录
- 再使用脚本定位二级或三级目录,成功率会明显更高

**补充说明**:
- 当前版本包含本地调试日志逻辑,会尝试向 `http://127.0.0.1:7777/event` 发送调试信息
- 如果你的电脑上没有启动这个本地调试服务,请求会自动失败并被忽略,不会影响正常使用

### ❌ 点确定后转存失败

这一步是**百度自己的逻辑**,和脚本无关。常见原因:
- 分享已失效 / 提取码错
- 目标目录已经有同名文件
- 网盘容量不够
- 分享者删了文件

## 七、脚本能做什么、不能做什么

**能**:
- ✅ 在分享页转存弹窗顶部注入搜索框
- ✅ 递归扫描你自己网盘的**文件夹结构**(不含文件),缓存到本地
- ✅ 模糊搜索目录名 + 全路径
- ✅ 最近使用的目录置顶
- ✅ 点结果后模拟操作原生弹窗跳过去

**不能**:
- ❌ 绕过百度网盘的下载限速(那是另一类脚本)
- ❌ 突破分享次数限制、提取码等
- ❌ 免登录搜别人网盘
- ❌ 自动"确定"转存(要你自己看清楚再点)

## 八、隐私说明

- 所有 API 请求都在**你自己的浏览器里**发,直接调百度网盘现有的 `/api/list`,和你手动浏览网盘时百度看到的请求一模一样
- 索引数据(你自己的文件夹路径列表)**只存在本地** Tampermonkey 存储里
- 当前调试版只会尝试把调试日志发到你自己电脑上的 `127.0.0.1`,不会发送到外部服务器; 如果本机没启动调试服务,这些请求会直接失败并被忽略

## 九、常见问题

**Q:这脚本会不会导致我账号被封?**

理论上比手动浏览"请求量大一点"。默认配置(concurrency=4, 每次翻页 1000 条)对于普通规模网盘一次全量扫描发几十到几百个请求,和你正常用网盘的差别不大。**别改到 concurrency=20+ 这种就好**。

**Q:能不能改成阿里云盘 / 夸克网盘用?**

不能直接用。它们的 API、DOM 完全不一样。要另写一份。

**Q:换了电脑要重新建索引吗?**

要。索引存在 Tampermonkey 本地存储里,不跨设备。换电脑重装脚本后,首次打开分享页会自动重新扫。

**Q:我改了网盘目录后,索引没更新怎么办?**

24 小时内会用旧缓存。手动点"刷新索引"立即重建。

### 本次故障复盘

#### 1. 为什么会出现"已定位到 xxx",但实际上没有跳到对应目录

- 旧逻辑里,`navigateDialogToPath()` 只要循环找到了每一级段名,最后就会直接显示"已定位到"
- 但百度网盘弹窗里"找到同名节点"和"真正切换到该目录"不是一回事
- 某些节点只是被匹配到了文本,点击后并没有触发目录切换,或者切换发生在错误的容器里
- 所以会出现提示成功,但原生弹窗路径没变的假成功

**现在的修复方式**:
- 每点击一级目录后,脚本都会额外校验:
  - 当前弹窗路径文本里是否已经出现预期路径
  - 或者下一层目标节点是否已经出现在当前目录树中
- 如果校验没通过,就继续尝试展开/切换
- 最终路径仍然没切过去时,直接报失败,不再给出误导性的成功提示

#### 2. 为什么会出现"定位失败,未找到目录节点"

- 从截图里的日志可以看到,旧逻辑会在错误范围内扫描出大量可见节点
- 这是因为脚本当时会把整个弹窗 body、内容区甚至其他列表区域也一起作为"目录树候选容器"
- 一旦扫描范围过大,就容易出现:
  - 当前层该找的节点其实没在真正目录树里渲染出来
  - 但脚本仍在别的容器里反复滚动和匹配
  - 最后连续重试很多次后仍报"未找到目录节点"

**现在的修复方式**:
- 优先给目录树容器打分,更偏向真实的文件树区域
- 节点收集时过滤掉明显不是目录行的长文本/杂项元素
- 减少无效重试轮次,让失败更快暴露,日志也更干净

#### 3. 为什么这次还会剩下"一级目录必须手动展开"的问题

- 这次虽然已经把日志链路补齐了,但从现象上看,`全能视频` 这类一级目录的点击并不总能让百度树组件进入稳定的展开态
- 当前比较像下面两种情况之一:
  - 点击事件发出去了,但百度内部没有把这次点击认成真正的"展开目录"
  - 子节点文字已经出现在 DOM 里,但没有稳定进入脚本的候选节点集合,导致后续匹配仍然找不到
- 所以这次版本的结果不是"彻底修复",而是先把根因范围收窄到了"展开态建立失败"和"展开后子节点收集失败"这两类

**当前结论**:
- 二级定位逻辑本身不是完全失效
- 真正卡住的是一级目录的自动展开闭环还不稳定
- 只要用户手动把一级目录先展开,后续二级目录定位通常能正常进行

**后续还需要继续做的事**:
- 继续观察 `expand_start` / `expand_after` / `activate_after_click` 这几类日志
- 重点确认百度目录树到底依赖哪个真实交互目标才能进入展开态
- 如果确认是候选节点过滤过严,再针对 `collectTreeNodes()` / `isUsableTreeNode()` 做最小修复
